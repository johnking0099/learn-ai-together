# 02 - 部署与服务化：知道"怎么跑"

模型训练好之后只是一堆文件（权重），要变成用户能调用的 API，中间需要经历"部署与服务化"。这是 PM 最需要理解的技术环节。

---

## GPU 基础：推理的"发动机"

大模型推理几乎都依赖 GPU。你需要了解的核心概念只有一个：**显存（VRAM）**——GPU 上的专用内存，决定了能装下多大的模型。

| GPU 型号 | 显存 | 典型定位 | 大致价格区间（云端/小时） |
|----------|------|---------|------------------------|
| A10G | 24 GB | 小模型推理 | $1-2 |
| L40S | 48 GB | 中等模型推理 | $2-4 |
| A100-80G | 80 GB | 主力推理卡 | $3-5 |
| H100-80G | 80 GB | 旗舰级，更快 | $5-8 |

**直觉类比**：显存就像车的后备箱——模型装不下就跑不了。而 GPU 的算力就像发动机马力——决定了跑多快。

---

## 多卡并行：一个模型，多张 GPU

当模型太大，一张 GPU 装不下时，需要拆分到多张卡上。

| 并行方式 | 做什么 | PM 需要知道的 |
|----------|-------|--------------|
| Tensor Parallelism (TP) | 把模型的每一层切开，分到多张 GPU | 最常用，降低单次推理延迟 |
| Pipeline Parallelism (PP) | 把模型的不同层分到不同 GPU | 配合 TP 使用，适合超大模型 |
| Data Parallelism (DP) | 多个模型副本处理不同请求 | 提高吞吐量，适合高并发 |

**对产品意味着什么**：TP=2 意味着一个模型实例至少需要 2 张 GPU——直接影响成本。PM 需要理解"用几张卡"和"服务几个用户"之间的关系。

> 延伸阅读：vLLM 的并行配置参数见 [`vllm-help/docs/5-1-core-parallel.md`](../vllm-help/docs/5-1-core-parallel.md)

## 推理框架：模型的"运行环境"

推理框架是把模型文件变成可服务的 API 的软件。主流选择：

| 框架 | 特点 | 适用场景 |
|------|------|---------|
| **vLLM** | PagedAttention、高吞吐、OpenAI 兼容 API | LLM 推理的首选 |
| **SGLang** | 高性能、RadixAttention | 与 vLLM 竞争的新锐 |
| **TGI** | HuggingFace 出品，集成好 | HuggingFace 生态用户 |
| **TensorRT-LLM** | NVIDIA 官方、极致性能 | 对性能要求极高的场景 |

**对产品意味着什么**：框架选型影响支持的模型范围、API 格式、性能上限。目前 vLLM 是社区最活跃的选择。

---

## 关键性能指标

作为 PM，你需要关注这几个核心指标：

| 指标 | 全称 | 含义 | 用户感知 |
|------|------|------|---------|
| **TTFT** | Time To First Token | 用户发送请求到看到第一个字的时间 | "开始响应快不快" |
| **TPS** | Tokens Per Second | 每秒生成的 token 数 | "吐字速度快不快" |
| **Throughput** | 吞吐量 | 系统每秒处理的总 token 数 | 平台侧关注的并发能力 |
| **并发数** | Concurrency | 同时服务的请求数量 | "能同时服务多少人" |

**TTFT 和 TPS 是用户体验的核心指标**。用户不会说"吞吐量低"，但会说"太慢了"——所以 PM 需要能把性能指标翻译成用户体验语言。

---

## KV Cache：LLM 推理的内存瓶颈

LLM 在生成文本时，需要"记住"已经处理过的所有 token 的中间状态，这些状态就存在 KV Cache 里。

**为什么重要**：
- KV Cache 是推理过程中**显存占用最大的部分**（通常超过模型本身）
- 上下文越长、并发越多，KV Cache 越大
- 显存不够 → 要么降低并发数，要么触发抢占（preemption），影响用户体验

**Prefix Caching**：如果多个请求共享相同的前缀（如系统提示词），可以复用 KV Cache，节省计算和显存。这对"千人千面但系统提示词相同"的场景非常有价值。

> 延伸阅读：vLLM 的 KV Cache 配置见 [`vllm-help/docs/3-1-core-memory.md`](../vllm-help/docs/3-1-core-memory.md)

## 调度策略：请求排队的学问

当并发请求超过 GPU 处理能力时，推理框架需要决定"先处理谁"。

- **Continuous Batching**：不等一批请求全部完成再处理下一批，而是有空位就立刻插入新请求——大幅提升吞吐量
- **抢占（Preemption）**：显存不够时，暂停某些请求的处理，腾出空间给其他请求
- **队列管理**：请求太多时如何排队、超时如何处理

**对产品意味着什么**：调度策略直接影响"用户等多久能得到回复"。PM 需要关注的是超时策略和用户等待体验，具体调度算法交给工程师。

> 延伸阅读：vLLM 的调度配置见 [`vllm-help/docs/4-1-core-scheduling.md`](../vllm-help/docs/4-1-core-scheduling.md)

## API 设计：模型对外的"接口"

主流推理框架都支持 **OpenAI 兼容接口**，这意味着：

- 前端/客户端可以用同一套 SDK 调用不同的模型
- 切换模型时不需要改业务代码
- 支持 **streaming**（流式输出）：用户一边看到文字一边生成，而不是等全部生成完才显示
- 支持 **structured output**（结构化输出）：强制模型输出符合 JSON Schema 的内容，方便下游处理

**对产品意味着什么**：OpenAI 兼容是现在的事实标准。如果你的平台提供 API 服务，兼容 OpenAI 格式几乎是必选项。

> 延伸阅读：vLLM 的 structured output 配置见 [`vllm-help/docs/9-structured-outputs.md`](../vllm-help/docs/9-structured-outputs.md)
